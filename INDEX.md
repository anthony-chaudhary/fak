## Measuring sticks (the scorecards)

Each turns a fuzzy goal into a number you can drive toward zero.

- [Industry scorecard](docs/industry-scorecard/README.md) — the one OUTWARD stick: an industry-first taxonomy of the dimensions the LLM-serving field competes on (vLLM/SGLang/TensorRT-LLM/llama.cpp), with fak's honest position on each (mostly named gaps). Two driven numbers: **coverage** (of the field) and **parity-debt** (honesty of the rows). Modular folder, regenerated from `tools/industry_scorecard.data/`.
- [Agent-readiness scorecard](docs/AGENT-READINESS-SCORECARD.md) — can an AI agent discover, adopt, and build on fak (friction-debt across the three steps an agent walks).
- [Repo-hygiene scorecard](docs/REPO-HYGIENE-SCORECARD.md) — verbosity, organization, indexing, and accessibility of the whole tree.
- [Code-quality scorecard](docs/CODE-QUALITY-SCORECARD.md) — the Go module's code-debt.
- [Steerability scorecard](docs/STEERABILITY-SCORECARD.md) — the one GROWTH-INVARIANT stick: does the effort to steer/change/navigate the repo stay flat as it grows? A 0–100 **steerability index** over modularity, coupling (blast radius), navigability, and correction — every KPI a ratio/percentile, so a 2× repo with the same discipline scores the same. The number every count-based sibling can't give.
- [Observability scorecard](docs/OBSERVABILITY-SCORECARD.md) — the observability plane (metrics, dashboards, alerts, traces, proofs, ship-audit), graded so every dashboard/alert/doc points at a metric the binary emits and every claim is verifiable.
- [Conflation scorecard](docs/CONFLATION-SCORECARD.md) — the truth-maintenance stick: does every number/status fak reports label its provenance — WITNESSED (a fact fak authored, in its control) vs OBSERVED (a value fak relays from an external party) — and never blame a provider-side miss on a fak action? Reads the fact-reporting surfaces (metric help, the guard exit summary) so a dashboard can't say "fak broke the cache" when the provider's cache simply expired.
- [Demo-quality scorecard](docs/DEMO-QUALITY-SCORECARD.md) — are the demos runnable, honest, self-contained.
- [Demo-robustness scorecard](docs/DEMO-ROBUSTNESS-SCORECARD.md) — do the demos survive bad input and odd environments.
- [Learning-docs scorecard](docs/LEARNING-SCORECARD.md) — does the teaching set actually teach (learning-debt: a how-to with no runnable command, a tutorial with no worked output, an orphan lesson, an uncovered topic). Pedagogy counterpart of repo-hygiene (structure) and doc-appeal (voice).
- [Code-2x program](docs/CODE-2X-PROGRAM.md) — the plan to halve code-debt, then halve it again.

- [Fusion, ref-counting, and GC for the agent loop (2026-06-25)](docs/notes/FUSION-REFCOUNT-GC-FOR-THE-AGENT-LOOP-2026-06-25.md) — the runtime-heap lens on the loop: every turn and tool result is a **heap object**, a pinned goal / active slash-command / open task is a **GC root**, compaction is **garbage collection**. The decisive finding after mapping the tree: the collector machine already ships at the byte/KV depths (`abi.CASPinner` refcount-by-digest, `ProvisionalSink.Promote/Rollback` commit-or-collect, `ctxplan.Forecast.Pins` whose own doc names "the active goal" as the canonical pin, `cachemeta.Lifecycle` TTL+`Touch`-revive) — what is missing is the single **edge from an intentional root to the heap objects it keeps alive**. Locates the gap exactly at `SessionPlanner.pins()` (the complete current root-set is system/first-user/last-user; the goal is absent), defines the **ref-count of a turn** (its live referents: an open goal, a later citing turn, a pending consumer), the conservative over-retain safety asymmetry (a false-free silently loses needed context; a false-retain costs a few tokens), **turn-fusion** (collapse a refcount-1 producer→consumer intermediate, the sound version of `maybeCompactInboundTools`) and **command-tenuring** (a recurring slash-command promotes to a compact rollup — the "store it for aggregate/GC" the goal asked for), and a 4-step first build that wires the goal into the per-turn pin set and measures the effect on the shipped signal/noise metric.
- [vToolcall, forked — best-effort serve-tiers and owning the loop (2026-06-25)](docs/notes/VTOOLCALL-FORK-AND-BESTEFFORT-2026-06-25.md) — the forward-looking sequel to the materialized-view note, answering "best effort" + "fork an open harness (Crush / opencode)". Adds (1) a **consistency-level QoS axis** (`STRICT | BOUNDED_STALE | BEST_EFFORT | SPECULATIVE` threaded as `c.Meta["consistency"]`, a DB-read-consistency dual) with a per-topology graceful-degradation capability matrix, and (2) the **fork direction**: make fak the tool *dispatcher* so the prior note's "impossible on the proxy" verdict flips and serve-as-ran runs closed-loop on real built-in tools. Honest spine, all three stated plainly: forking does not *soften* un-ring, it **relocates** it (after-consumption → a before-consumption write barrier fak owns, needing a net-new suspend-and-resume turn primitive — today's `gateTurn` only terminates); **Crush is FSL-1.1-MIT** (source-available, commercial-restricted at HEAD) so the license-clean call is **build the minimal native TUI on the shipped `internal/agent` seam** (Apache-uniform, single-binary) or host **opencode-as-server** (MIT), never fork Crush into the Apache module; and the loop fak would extend is a **benchmark A/B comparator (`RunArm`) today**, not a production loop. Portfolio verdict: keep the proxy as the universal gate, lead the fork with the native build.
- [vToolcall — tool results as a materialized view (2026-06-25)](docs/notes/VTOOLCALL-MATERIALIZED-VIEW-2026-06-25.md) — the *closed-loop* dual of [vCache](docs/notes/VCACHE-VIRTUAL-API-CACHE-2026-06-24.md): serve a tool result from the kernel's own cache **as if the tool ran**, framed as one materialized view in a query engine over the whole harness-facing surface (tool-defs via `promptmmu`, history via `agent.CompactAnthropicHistory`, context via `contextq`, KV via vCache — and now results via the vDSO). The serve mechanism already ships as the vDSO FastPath (`kernel.go:347` → `abi.FastPaths().Lookup`); this note is its design rationale with **cache invalidation as the spine**. Honest build boundary: result-synthesis is real only where fak *dispatches* the tool (`fak_syscall`/in-kernel), structurally impossible on the `/v1/messages` proxy (the harness runs tools there); witness-keying of the tier-2 key, a cross-process FILL pool, and the speculative-serve `ProvisionalSink` are named **unbuilt**. Law T (sound-before-serve) is the polarity flip from vCache's A2 — a stale tool result *corrupts*, it does not merely *cost*.
